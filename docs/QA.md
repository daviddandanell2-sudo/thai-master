# QA

## Purpose

Quality assurance, testing, validation, review, and acceptance criteria for the project.

This document defines what "done" means for every type of work.

## Page QA Checklist

### Before QA
- Page has been built by Builder Worker
- Images have been generated by Image Worker
- Page is saved in `/content/`

### Automated Checks
- [ ] Word count: 800+ (service pages) or 1200+ (blog)
- [ ] H1 present and includes primary keyword
- [ ] Meta description: 120-160 characters
- [ ] Only one H1 per page
- [ ] H2 and H3 tags used logically
- [ ] Primary keyword in first 100 words
- [ ] Internal links: 3+ with descriptive anchor text
- [ ] FAQ: 5-7 items
- [ ] CTA present
- [ ] No broken links
- [ ] Images have alt text
- [ ] Images follow `docs/image-rules.md`
- [ ] Filename is SEO-friendly

### Manual Checks
- [ ] Content is useful and answers a real parent question
- [ ] Keyword intent is clear
- [ ] Page is specific (not generic city-swap)
- [ ] Page is different from similar pages
- [ ] No fake claims, testimonials, or credentials
- [ ] Brand voice matches `docs/brand-voice.md`
- [ ] Tone is calm, helpful, not salesy
- [ ] Page helps parents make a decision

### Image Verification
- [ ] Tutor is Thai
- [ ] Child is European and alone
- [ ] One child only (never two, never zero)
- [ ] Private setting (apartment/villa)
- [ ] No classroom or school background
- [ ] Premium and comfortable
- [ ] Image matches page topic

### Browser Review
- [ ] Page looks good in browser
- [ ] Images display correctly
- [ ] Text is readable
- [ ] Buttons are visible and clickable
- [ ] Mobile layout works correctly
- [ ] No horizontal scrolling on mobile
- [ ] Tap targets are appropriately sized

### Scoring

| Score | Rating | Action |
|---|---|---|
| 9 | Excellent | Ready to publish |
| 8 | Good | Minor fixes, then publish |
| 7 | Acceptable | Fixes required before publish |
| 6 or below | Poor | Major revision or discard |

## System QA Checklist

### Before System Changes
- Change has been tracked in `docs/TRACKING.md`
- Change aligns with `docs/STRATEGY.md`
- Risks have been checked in `docs/RISKS.md`

### Validation
- [ ] No files deleted without approval
- [ ] No `README.md`, `AGENTS.md`, or `CLAUDE.md` modified without approval
- [ ] JSON files are valid JSON
- [ ] Markdown files render correctly
- [ ] All cross-references are valid
- [ ] Git status is clean (or changes are documented)

## Deployment QA Checklist

### Before Deployment
- All pages have passed QA (score 7+)
- Sitemap is generated
- robots.txt is correct
- Tracking is ready

### Post-Deployment
- [ ] Pages are live
- [ ] URLs work correctly
- [ ] Images load
- [ ] CTAs work
- [ ] Mobile layout correct
- [ ] Sitemap submitted to Search Console
- [ ] Tracking firing correctly

## QA Roles

| Role | Responsibility |
|---|---|
| QA Worker | Run QA checks, browser review, scoring |
| Fixer Worker | Fix issues found by QA |
| Boss AI | Assign QA, review reports, trigger handoffs |
| Human | Approve publish decisions, handle escalations |

## Escalation

If a page fails QA 3 times:
1. Stop
2. Document failures
3. Escalate to Human via Boss AI
4. Do not proceed without Human decision
